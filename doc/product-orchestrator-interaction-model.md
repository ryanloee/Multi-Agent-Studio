# MAS 产品交互模型方案

> 目标：把 MAS 从“会聊天的开发工具”收敛成“可视化项目编排台”。
> 这份文档回答 5 个问题：
> 1. 用户如何进入系统
> 2. Planner 到底负责什么
> 3. 什么是唯一真实状态
> 4. 何时才能真正执行
> 5. 新项目和维护项目分别怎么用

---

## 1. 核心判断

当前系统最主要的问题，不是节点类型不够，也不是模型不够多，而是**心智模型不稳定**：

- 用户以为自己在“和 Planner 协商并开始干活”
- 系统有时把聊天当执行入口
- 画布、聊天、运行状态不是同一个真相源
- 新项目和维护项目的入口没有被统一解释

这会直接导致三个体验问题：

1. 用户不知道什么时候只是“讨论”，什么时候已经“开始执行”
2. 用户不知道维护老项目应该从哪一步开始
3. 用户不知道出问题时该看聊天、看画布、还是看任务面板

因此，后续开发前必须先统一产品逻辑。

---

## 2. 产品定位

MAS 不应该被定义成：

- 一个能聊天的 IDE 插件
- 一个能自动开跑的黑盒 Agent
- 一个预设“开发 / 修 Bug / 巡检 / 重构”模式的流程机

MAS 更准确的定位应该是：

**一个以 DAG 画布为核心、由 Planner 辅助规划、由用户明确批准执行的项目编排台。**

关键词：

- 白盒
- 可视化
- 可审阅
- 可恢复
- 可追踪

一句话定义：

**用户用自然语言表达目标，Planner 把目标整理成可执行 DAG，用户确认后系统执行当前 DAG，并把结果回流到画布和任务系统。**

---

## 3. 唯一真相

### 3.1 当前画布 DAG 是唯一真实状态

必须明确：

- Planner 聊天内容不是可执行真相
- 某条“开始”消息不是可执行真相
- 某个任务面板摘要也不是可执行真相

唯一真实状态是：

**当前工作流画布上的节点、边、配置、模型、工作目录。**

执行时系统只读这一份状态。

### 3.2 三层分离

系统应分成三层：

1. **意图层**
   - 用户输入目标、问题、约束
   - Planner 负责理解和组织

2. **结构层**
   - 画布上的 DAG
   - 节点、边、模型、依赖、工作目录

3. **执行层**
   - Run snapshot
   - 沙箱
   - 任务
   - 产物
   - 日志

意图层可以改结构层，结构层可以触发执行层，但三者不能混淆。

---

## 4. 统一流程模型

不建议把系统切成“开发模式 / 修 Bug 模式 / 巡检模式 / 重构模式”。

更自然的方式是采用统一阶段流：

1. `Intake`
2. `Assess`
3. `Plan`
4. `Ready`
5. `Run`
6. `Review`
7. `Iterate`

### 4.1 阶段定义

#### Intake

用户表达想做什么。

典型输入：

- 做一个游戏 CDK 售卖网站
- 修复支付回调失败的问题
- 看看这个项目怎么启动
- 给现有后台增加批量导入功能

系统职责：

- 接收任务对象
- 收集项目路径、已有上下文、项目文档

#### Assess

系统先理解现状，而不是急着编码。

典型动作：

- 扫描项目结构
- 识别技术栈
- 识别启动方式
- 识别关键目录和依赖
- 生成“当前项目摘要”

这一步对维护项目尤其关键。

#### Plan

Planner 根据目标和现状生成或修改 DAG。

典型动作：

- 任务拆分
- 依赖关系建立
- 子节点模型分配
- 节点 prompt 生成
- 合并策略确定

#### Ready

当前 DAG 已具备执行条件，但还没有真正运行。

这一步是用户体验上最关键的一层缓冲：

- 用户看到方案已经准备好
- 但系统还没有产生副作用
- 用户可以检查节点和工作目录

#### Run

点击运行按钮后，系统对**当前 DAG 快照**执行。

执行必须是明确动作，不能从聊天中隐式触发。

#### Review

执行结束后，用户看：

- 节点状态
- diff
- 产物
- 日志
- 测试结果
- merge 报告

#### Iterate

用户基于 review 结果继续调整 DAG，再进入下一轮。

## 5. 工作流状态机

建议工作流层面明确 5 个主状态：

- `Draft`
- `Ready`
- `Running`
- `Blocked`
- `Review`

### 5.1 状态含义

#### Draft

还在讨论或构建中。

特征：

- DAG 可能不完整
- 工作目录可能未设置
- Planner 还在改图

#### Ready

方案已可执行，但尚未运行。

特征：

- DAG 已稳定
- 校验通过
- 等待用户点击运行

#### Running

当前 DAG 快照正在执行。

特征：

- 运行状态和编辑状态分离
- 画布显示的是当前工作流
- 执行的是某次 snapshot

#### Blocked

系统知道不能继续执行，并明确告诉用户缺什么。

典型原因：

- 未设置工作目录
- 模型未配置
- DAG 没有可执行节点
- merge 冲突待处理
- 人工审批未完成

#### Review

一轮执行结束，等待用户审阅结果并决定下一步。

### 5.2 允许的状态流转

```text
Draft -> Ready
Draft -> Blocked
Blocked -> Draft
Blocked -> Ready
Ready -> Running
Running -> Review
Running -> Blocked
Review -> Draft
Review -> Ready
```

不允许：

- Draft 直接自动进入 Running
- 聊天消息直接改变为 Running

---

## 6. Planner 的职责边界

### 6.1 Planner 应该负责什么

Planner 负责：

- 理解目标
- 理解当前项目现状
- 生成 DAG
- 修改 DAG
- 分配依赖关系
- 给出执行建议
- 判断当前方案是否进入 `Ready`

### 6.2 Planner 不应该负责什么

Planner 不应该：

- 直接启动执行
- 把“开始”当成系统副作用命令
- 在没有落到画布前就认为方案已生效
- 越过用户直接改变运行状态

### 6.3 Planner 的正确输出

Planner 的输出应该始终服务于画布和状态机，而不是代替运行按钮。

一轮 Planner 输出应该包含两部分：

1. 面向人的说明
2. 面向系统的结构化动作

建议结构化动作至少支持：

- `update_dag`
- `set_ready`
- `ask_clarification`
- `report_blocker`

注意：

这里仍然**不建议**让 Planner 输出 `run_now`，否则又会把聊天和执行耦合起来。

---

## 7. Run 的唯一语义

### 7.1 运行按钮是唯一执行入口

系统必须统一成：

- 聊天可以修改计划
- 运行按钮才会真正执行

这条规则足够简单，也最符合可控系统的直觉。

### 7.2 点击运行时发生什么

点击运行后：

1. 读取当前工作流画布
2. 做执行前校验
3. 生成 DAG snapshot
4. 创建 run
5. 运行 snapshot

因此，执行对象永远不是“最新聊天消息”，而是“当前画布快照”。

### 7.3 Run 前校验清单

建议统一校验：

- 是否设置工作目录
- 工作目录是否存在
- 至少存在一个可执行节点
- 节点模型是否可解析
- Planner 生成的 DAG 是否闭合
- merge 依赖是否合法
- shell 节点是否具备必要命令上下文

校验失败不应偷偷继续，而是进入 `Blocked`。

---

## 8. 首页入口设计

不建议做“四种模式”。

建议首页只保留三个入口：

1. `新建目标`
2. `导入现有项目`
3. `处理一个任务 / 问题`

### 8.1 新建目标

适合：

- 从零做一个网站
- 做新功能
- 做原型

默认行为：

- 新建空工作流
- 引导输入目标
- Planner 从目标生成 DAG

### 8.2 导入现有项目

适合：

- 接手陌生仓库
- 给老项目加功能
- 分析现有代码

默认行为：

- 先要求设置工作目录
- 自动进入 `Assess`
- 先生成项目摘要，再进入 Planner

### 8.3 处理一个任务 / 问题

适合：

- 修一个 Bug
- 跟踪一个线上问题
- 处理一个明确需求

默认行为：

- 输入 issue / 任务描述
- 允许附带项目路径
- 先做局部 Assess，再出 DAG

这三种入口覆盖面已经足够大，而且不强迫用户先给自己归类。

---

## 9. 新项目与维护项目的差异

### 9.1 新项目工作流

新项目默认链路：

```text
目标输入 -> Planner 拆解 -> 生成 DAG -> 用户检查 -> Run
```

重点在：

- 规划
- 结构搭建
- 并行编码
- 合并和验证

### 9.2 维护项目工作流

维护项目默认链路：

```text
选择工作目录 -> Assess 当前项目 -> 形成项目摘要 -> Planner 拆解变更 -> Run
```

重点在：

- 理解现状
- 缩小影响面
- 只修改必要部分
- 强调 review 和验证

### 9.3 为什么维护项目不能直接套“开发模式”

维护项目最怕的是：

- 一上来就重构过多
- 没读懂现有依赖
- 不知道怎么启动
- 不知道改动影响范围

所以维护项目一定要显式有 `Assess` 阶段。

## 10. 建议的主界面结构

### 10.1 总体布局

建议主界面固定四区：

1. 左侧：任务对象 / 项目文档 / 项目摘要
2. 中间：当前 DAG 画布
3. 右侧：节点 / 工作流配置
4. 底部：执行过程与白盒信息

### 10.2 左侧区域职责

左侧不只是节点库，还应该承担“项目理解上下文”：

- 当前目标
- 项目文档
- Assess 结果
- 任务列表
- 最近 artifact

### 10.3 中间画布职责

画布是结构层：

- 显示当前 DAG
- 提供 Ready 前最后确认
- 成为唯一可执行真相

### 10.4 底部面板职责

底部是执行层：

- LLM 输出
- 工具调用
- Shell 输出
- Timeline
- Planner 对话

聊天应该在这里作为“规划辅助”，而不是主舞台。

---

## 11. “开始”这类自然语言怎么处理

这是最容易让系统变怪的地方。

### 11.1 不建议的做法

- 前端正则匹配“开始/运行”直接执行
- Planner 看到“开始”就自动开跑
- 聊天和运行按钮并存两个执行入口

### 11.2 建议的做法

用户在 Planner 里说“开始”，系统理解为：

**“请确认当前方案是否已经可执行。”**

然后 Planner 的合理反馈应该是：

- 如果还不完整：继续补方案
- 如果完整：把状态设为 `Ready`
- UI 明确提示：`方案已就绪，请点击顶部运行按钮开始执行`

这样更符合人类直觉：

- 用户没有被“忽略”
- 系统也没有偷偷执行
- 执行入口仍然唯一

## 12. 对当前系统的落地改造建议

### P0：先修产品逻辑

1. 去掉聊天中自动触发运行的机制
2. 引入 `Draft / Ready / Running / Blocked / Review` 状态
3. 明确“画布 DAG 是唯一真相”

### P1：补齐维护项目体验

1. 增加 `Assess` 阶段
2. 导入现有项目时自动扫描结构
3. 生成项目摘要并落到项目文档

### P2：统一 Planner 协议

1. Planner 输出结构化动作
2. 前端只根据结构化动作更新 DAG 和状态
3. Planner 不再承担执行入口语义

### P3：优化 UI 提示

1. 在顶部显示当前工作流状态
2. 在 Ready 状态下高亮运行按钮
3. 在 Blocked 状态下明确缺项
4. 在 Review 状态下强调结果入口

---

## 13. 最终交互原则

后续所有功能开发都建议遵守这 7 条原则：

1. 画布 DAG 是唯一可执行真相
2. Planner 只负责规划，不负责执行
3. 运行按钮是唯一执行入口
4. 任何副作用都必须发生在明确执行之后
5. 维护项目必须先经过 Assess
6. Review 是一等阶段，不是附属日志
7. 不让用户先选模式，只让系统识别当前阶段

---

## 14. 一句话版本

MAS 最自然的形态不是：

> “选一个模式，然后和 Agent 聊天帮你写代码。”

而应该是：

> “给系统一个目标或一个项目，它先理解现状，再把方案变成画布，你确认后再运行，并在白盒面板里审阅整个执行过程。”
